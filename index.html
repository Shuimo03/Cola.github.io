Restful API使用(一)

Restful API使用(一)

简介

REST 是一种基于超媒体构建分布式系统的架构风格。 REST 独立于任何基础协议，并且不一定绑定到 HTTP。 但是，最常见的 REST 实现使用 HTTP 作为应用程序协议。

基于 HTTP 的 REST 的主要优势在于它使用开放标准，不会绑定 API 的实现，也不会将客户端应用程序绑定到任何具体实现。

RESTful API主要原则

  1. RESTful API围绕资源进行设计，资源可以由客户端访问任何类型的对象、数据或者服务。
  2. 每个资源都一个标识符，能够唯一标识该资源的URL。
  3. 客户端通过交换资源的表达方式来和服务交互，主流的是JSON作为交换格式。
  4. RESTful API使用统一接口，这有助于分离客户端和服务实现，对于基于HTTP构建的 RESTful API，统一接口包括使用标准的HTTP谓词对资源执行操作，最常见的是GET、POST、PUT、PATCH、DELETE。
  5. 资源名用复数形式表示
  6. REST API 使用无状态请求模型。 HTTP 请求应是独立的并可按任意顺序发生，因此保留请求之间的瞬时状态信息并不可行。 信息的唯一存储位置就在资源内，并且每个请求应是原子操作。 此约束可让 Web 服务获得高度可伸缩性，因为无需保留客户端与特定服务器之间的关联。 任何服务器可以处理来自任何客户端的任何请求。

CRUD对应的HTTP标准操作

对于资源的具体类型，由HTTP动词来决定，常用的HTTP动词有以下五个(括号内是对应的SQL命令)

  1. GET（SELECT）：从服务器取出资源（一项或多项）。
  2. POST（CREATE）：在服务器新建一个资源。
  3. PUT（UPDATE）：在服务器更新资源（客户端提供改变后的完整资源）。
  4. PATCH（UPDATE）：在服务器更新资源（客户端提供改变的属性）。
  5. DELETE（DELETE）：从服务器删除资源。

分布式系统原理与范型chapter1

分布式的定义

分布式经典的一种定义是，多台计算机的集合，这个集合对于用户来说是一个单机系统，好比计算机内部结构，对于普通用户来说，计算机就是一个整体，实际上计算机是由不同的硬件组成。

分布式必须能够让用户方便访问资源，必须隐藏一个资源在一个网络上分布的一个事实，必须是开发的，而且是可扩展的。

分布式的特性

  1. 透明性
    1. 分为透明性
    2. 位置透明性
    3. 迁移透明性
    4. 重定位透明性
    5. 赋值透明性
    6. 并发透明性
    7. 故障透明性
  2. 可扩展性
    1. 隐藏通信等待时间
    2. 分布技术
    3. 复制技术
  1. 共享性
  2. 开放性
    1. 互操作性
    2. 可移植性

可扩展性遇到的问题

当一个系统需要进行扩张的时候，首先要考虑规模上的扩展，在需要支持更多用户货资源的时候，常常受到集中的服务、数据以及算法所造成的限制。
比如

  1. 集中式服务
  2. 集中式数据
  3. 集中式算法

分布式系统的类型

  1. 集群计算系统
  2. 网格计算系统

分布式信息系统

(ACID)事务处理的属性

  1. 原子性
  2. 一致性
  3. 独立性
  4. 持久性

分散算法

分散算法和集中式算法不同的特性:

  1. 没有任何计算机拥有关于系统状态的而信息
  2. 计算机只根据本地信息做出决策。
  3. 某台计算机的故障不会让算法崩溃。
  4. 没有这样的假设：存在全局性时钟。

docker部署springboot项目

因为我用的是IDEA的插件完成部署，不过大概步骤是可以借鉴的。

首先在IDEA中安装docker插件:方式为：File ——> Settings ——> Plugins，安装后重启IDE

第二步，登录服务器，修改docker文件配置，前提是已经安装好了docker。

找到ExecStart，在末尾加上-H tcp://0.0.0.0:2375，表示打开2375端口，并且支持远程连接docker。

修改完文件之后，需要重新加载配置：

  • systemctl daemon-reload
  • systemctl restart docker.service

IDEA中配置docker主机，在file ——> Settings ——> Docker
配置tcp连接，tcp://服务器ip:2375。

题海战术与刷题

通过之前一篇文章，关于如何抄代码的，以及这段时间刷题，发现了一些方法其实是可以通用的，比如说题海战术和刷题。

这里用leetcode和课后习题来做例子，如果一道算法题在5-10分钟之内没有思路，那就去看题解，或者看解题思路，这一步是找出自己之前的想法的错误所在，然后纠正这个错误，接着在15-20分钟去实现，如果实现不了，说明基础不够扎实，再去看答案，但是要把题目记下来，之后多刷几次，保证弄懂。

同理平时做的作业也是，面试题也是，如果实在没有思路就不要死磕和钻牛角，直接去看解题思路就行了，然后再写答案，一味的追求数量不追求数量，是没有用的，缺少思考的过程抄的答案和代码都属于无情的copy机器。
