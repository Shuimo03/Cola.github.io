(一)序言-SQL编译器

前言

上个学期本来打算写一个关系数据库来作为毕设的，但是懒癌和拖延症发作，就拖到了这个学期，正好又开了编译原理这门课，就可以顺道把SQL的编译器来完成。

关于这个系列

这个系列主要还是注重编译器的前端，同时记录编译原理这门课的过程，前期使用antlr4来完成Lexer和Grammar，这样可以减少工作量，打算等全部完成之后，再重新手写一个编译器。

antlr4介绍

这里引用一下官方的介绍就是:

ANTLR (ANother Tool for Language Recognition) is a powerful parser generator for reading, processing, executing, or translating structured text or binary files. It’s widely used to build languages, tools, and frameworks. From a grammar, ANTLR generates a parser that can build and walk parse trees.

简单来说就是，antlr是一个强大的语法生成器工具，可以用来读取，生产，执行和翻译二进制文件。

你可以用它来实现编译器的前端(Lexer和Grammar).

Lexer 词法分析器

一个词法分析器主要任务是读入源代码的字符，然后将它们组成词素(token)，然后生成并输出一个词法单元序列，每个此法单元对应一个词素，这个词法单元序列被输出到词法分析器进行语法分析，词法分析器还需要和符号表进行匹配。

当词法分析器发现一个标识符的词素的时候，还需要将这个词素添加到符号表里面。

简单来说就是，当我们平时把代码写好，运行的时候，词法分析器会先去读取代码中每一个字符，通过某种匹配算法，来检查这个是关键字还是其他字符。

这里使用ANTLR4来编写一个词法分析器，需要新建一个LexerFileName.g4，代码文件名和lexer grammar同名，在这个文件里面可以自己定义语法规则，最后通过anltr4 LexerFileName.g4 来生成三个文件，分别是:

  1. LexerFileName.java
  2. LexerFileName.interp
  3. LexerFileName.token

LexerFileName.g4只是一个语法规则文件，想要继续实现这个词法解析器，去把LexerFileName.java中内容实现了。

记录一次服务器被当矿机解决方法

腾讯云之前发来过警告说服务器被攻击了，当时是在tmp目录下，有一个文件无法删除，chmod 777 修改文件都没有办法，还好那个时候没有安装什么东西，于是就重装了，最近跑了一个docker服务，发现每次都会无缘无故多出两个镜像，一个是ubuntu，一个是其他的，后来每次自己的镜像总会消失，并且CPU占用率就居然有90%。。
发现是一个kdevtmpfsi进程，通过Google发现是一个挖矿病毒。。

解决方案

  1. 使用top命令，查看那个程序占用率CPU最高，一般是50%或者90%，接着使用kill -9 PID 杀掉这个进程，但是还会回来，是因为写了定时任务。
  2. 接着使用find / -name kdevtmpfsi和find / -name kinsing，找到这些文件，并且删除，这里有一个坑的地方在于，可能根本删不掉，就算是root权限都不行，可以使用chattr工具。
  3. crontab -l查看定时任务，如果这里有权限的话，就可以crontab -e把定时任务删除。

防护措施

主要原因还是因为对docker不熟悉，刚刚接触上手，这个病毒主要是通过扫描docker端口2375，可以在docker.service，把-H 0.0.0.0:2375改了，或者是在云服务器中的安全组里面关闭端口，同时修改ssh端口。

Restful API使用(一)

Restful API使用(一)

简介

REST 是一种基于超媒体构建分布式系统的架构风格。 REST 独立于任何基础协议，并且不一定绑定到 HTTP。 但是，最常见的 REST 实现使用 HTTP 作为应用程序协议。

基于 HTTP 的 REST 的主要优势在于它使用开放标准，不会绑定 API 的实现，也不会将客户端应用程序绑定到任何具体实现。

RESTful API主要原则

  1. RESTful API围绕资源进行设计，资源可以由客户端访问任何类型的对象、数据或者服务。
  2. 每个资源都一个标识符，能够唯一标识该资源的URL。
  3. 客户端通过交换资源的表达方式来和服务交互，主流的是JSON作为交换格式。
  4. RESTful API使用统一接口，这有助于分离客户端和服务实现，对于基于HTTP构建的 RESTful API，统一接口包括使用标准的HTTP谓词对资源执行操作，最常见的是GET、POST、PUT、PATCH、DELETE。
  5. 资源名用复数形式表示
  6. REST API 使用无状态请求模型。 HTTP 请求应是独立的并可按任意顺序发生，因此保留请求之间的瞬时状态信息并不可行。 信息的唯一存储位置就在资源内，并且每个请求应是原子操作。 此约束可让 Web 服务获得高度可伸缩性，因为无需保留客户端与特定服务器之间的关联。 任何服务器可以处理来自任何客户端的任何请求。

CRUD对应的HTTP标准操作

对于资源的具体类型，由HTTP动词来决定，常用的HTTP动词有以下五个(括号内是对应的SQL命令)

  1. GET（SELECT）：从服务器取出资源（一项或多项）。
  2. POST（CREATE）：在服务器新建一个资源。
  3. PUT（UPDATE）：在服务器更新资源（客户端提供改变后的完整资源）。
  4. PATCH（UPDATE）：在服务器更新资源（客户端提供改变的属性）。
  5. DELETE（DELETE）：从服务器删除资源。

分布式系统原理与范型chapter1

分布式的定义

分布式经典的一种定义是，多台计算机的集合，这个集合对于用户来说是一个单机系统，好比计算机内部结构，对于普通用户来说，计算机就是一个整体，实际上计算机是由不同的硬件组成。

分布式必须能够让用户方便访问资源，必须隐藏一个资源在一个网络上分布的一个事实，必须是开发的，而且是可扩展的。

分布式的特性

  1. 透明性
    1. 分为透明性
    2. 位置透明性
    3. 迁移透明性
    4. 重定位透明性
    5. 赋值透明性
    6. 并发透明性
    7. 故障透明性
  2. 可扩展性
    1. 隐藏通信等待时间
    2. 分布技术
    3. 复制技术
  1. 共享性
  2. 开放性
    1. 互操作性
    2. 可移植性

可扩展性遇到的问题

当一个系统需要进行扩张的时候，首先要考虑规模上的扩展，在需要支持更多用户货资源的时候，常常受到集中的服务、数据以及算法所造成的限制。
比如

  1. 集中式服务
  2. 集中式数据
  3. 集中式算法

分布式系统的类型

  1. 集群计算系统
  2. 网格计算系统

分布式信息系统

(ACID)事务处理的属性

  1. 原子性
  2. 一致性
  3. 独立性
  4. 持久性

分散算法

分散算法和集中式算法不同的特性:

  1. 没有任何计算机拥有关于系统状态的而信息
  2. 计算机只根据本地信息做出决策。
  3. 某台计算机的故障不会让算法崩溃。
  4. 没有这样的假设：存在全局性时钟。

docker部署springboot项目

因为我用的是IDEA的插件完成部署，不过大概步骤是可以借鉴的。

首先在IDEA中安装docker插件:方式为：File ——> Settings ——> Plugins，安装后重启IDE

第二步，登录服务器，修改docker文件配置，前提是已经安装好了docker。

找到ExecStart，在末尾加上-H tcp://0.0.0.0:2375，表示打开2375端口，并且支持远程连接docker。

修改完文件之后，需要重新加载配置：

  • systemctl daemon-reload
  • systemctl restart docker.service

IDEA中配置docker主机，在file ——> Settings ——> Docker
配置tcp连接，tcp://服务器ip:2375。
